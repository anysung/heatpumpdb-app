## Data Flow

```

        |
        v
   run-pricing.cjs                     attaches pricing layer
        |
        v

        |
        v
   enrich-physical-specs.cjs           attaches physical dimensions/weight
        |
        v
   output/dataset-enriched-full.json   (2) Consolidation source (6,514 items, all segments)
        |
        v
   consolidate-datasets.cjs            flattens, splits, adds manufacturer_short
        |
        +--> output/product-residential.json   (3) FINAL canonical â€” 4,387 residential
        +--> output/product-commercial.json    (3) FINAL canonical â€” 2,127 commercial
        +--> public/data/products.json         (4) App-serving copy (= product-residential.json)
```

Intermediate files between steps 1 and 2 are regenerated by the pipeline and
are not kept in the repository. Only the four numbered files above persist.

## When to Update `public/data/products.json`

Run `consolidate-datasets.cjs` whenever:
1. A new BAFA extraction is performed (new `bafa-luft-wasser.json`)
2. Pricing is re-run (`run-pricing.cjs`)
3. Physical specs are re-enriched (`enrich-physical-specs.cjs`)
4. Manufacturer short names are updated (`manufacturer-short-names.json`)

The script regenerates all three output files:
- `output/product-residential.json`
- `output/product-commercial.json`
- `public/data/products.json`

## Governance Rules

1. **Two canonical product files only.** `product-residential.json` and
   `product-commercial.json` are the final product datasets. Do not create
   additional competing product JSON files without documenting their role here.

2. **Do not edit `public/data/products.json` directly.** It is a generated
   copy. Changes belong upstream in the pipeline or in the consolidation script.

3. **Preserve internal segment values.** `residential_core`, `light_commercial`,
   and `commercial_project` are distinct validated segments. Do not collapse
   them into a single "commercial" value.

4. **Do not drop internal metadata.** Fields like `physical_specs_quarantined`,
   `segment_confidence`, and `review_flags` exist for traceability and QA.
   They must not be removed just because the UI does not currently display them.

5. **Document new enrichment layers.** If a new data layer is added to the
   pipeline, update `product-model-standard.json` and this file.

6. **Safety-check before consolidation.** Always audit the source fields
   against the keep list before generating final files. The v2.0 consolidation
   rescued 4 fields that would have been silently lost.

## Common Mistakes to Avoid

| Mistake | Why it's wrong | What to do instead |
|---------|---------------|-------------------|
| Editing `public/data/products.json` as the master dataset | It's a deployment copy, not the source of truth | Edit upstream data or config, then re-run `consolidate-datasets.cjs` |
| Using a deleted intermediate (e.g. `dataset-full.json`) as app data | It's a stale pipeline artifact with an older schema | Use `product-residential.json` or `product-commercial.json` |
| Dropping internal metadata fields to "simplify" | Loses traceability, QA, and future enrichment capability | Keep all fields defined in `product-model-standard.json` |
| Collapsing `light_commercial` + `commercial_project` into one segment | Destroys validated pricing and visibility distinctions | Preserve the three-value `market_segment` enum |
| Creating a new `dataset-xyz.json` without documenting it | Causes the same file proliferation problem this consolidation solved | Add it to this document with a clear role, or don't create it |
| Running only part of the pipeline and deploying | Produces files with mismatched schemas or missing fields | Run the full pipeline: pricing â†’ build â†’ enrich â†’ consolidate |
