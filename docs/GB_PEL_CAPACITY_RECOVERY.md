# GB — recovering a rated capacity for unmatched Ofgem PEL records

The Ofgem PEL publishes **no performance data at all** — no capacity, no COP, no
SCOP. A PEL record therefore only has a rated capacity if we can prove it is the
same hardware as a European product record. Without one it is `unclassified`: it
appears in neither Residential nor Commercial, and the count is disclosed on the
Products page (see `EUROPE_DATA_AND_PRODUCT_SEGMENTATION_PRINCIPLES.md` §2 — a
missing capacity is never quietly filed as residential).

In July 2026, 2,745 of 4,422 PEL heat pumps were unclassified. This pass recovered
**611** of them. The rest are still unclassified, on purpose.

* Rules: `scripts/ofgem/pel-match-lib.mjs` (pure, unit-tested)
* Runner: `scripts/ofgem/match-pel-recovery.mjs` (runs after the two primary matchers)
* Tests: `tests/pel-matching.unit.mjs`
* Detailed per-record output (gitignored, regenerated by the runner):
  `data_sources/ofgem_pel/matching/<snapshot>/pel-recovery-{matches,review,audit}.json`

---

## Why 2,745 records had failed

`match-pel-to-bafa` and `match-pel-to-eprel` compare the model as **one ordered
token sequence**. Every failure below follows from that:

| Cause | What it looks like |
|---|---|
| Component pair in the other order | PEL `ERGA04EAV3A / EHVH04SU18EA6V` vs EPREL `EHVH04SU18EJ6V / ERGA04EAV3A` |
| Package name prefixed to the unit | PEL `WiSAN-YME 1 S 2.1` vs registry `EDGE Evo 2.0 / WiSAN-YME 1 S 2.1` |
| Market suffix / descriptive phrase | PEL `WPC 05 S GB`, `… - UK Version - Heating Only` |
| One outdoor unit, several packages | The old rule demanded all ~30 copied spec fields be identical, so a **tank-size** difference discarded a record even when every candidate agreed on the capacity |
| Brand written with a city / legal-entity prefix | PEL `Guangdong New Energy Technology Co., Ltd.` vs registry `GD New Energy` |
| Brand genuinely absent from any European source | UK ground-source specialists (Kensa, Go Geothermal, Thermal Earth) and several Chinese OEMs |

## What the recovery pass does

Brand-gated throughout. Strongest stage first; a weaker stage never runs once a
stronger one has produced candidates.

1. **Exact model** — identical after formatting-only normalization. Tier A.
2. **Component identity** — one side's product codes wholly contain the other's
   (the candidate merely adds a package name or a tank). Tier A when unique.
3. **Shared unit code** — the records share a product code but neither contains the
   other. The outdoor unit is the same part, and the outdoor unit is what has a
   rated heat output. Tier B.
4. **Market-suffix family** — `WPC 05 S GB` → `WPC 05 S`, numeric tokens equal. Tier B.
5. **Fuzzy similarity** — produces a **review queue only**. It never accepts anything.

A "product code" means ≥6 characters containing both letters and digits
(`ERGA04DAV3A`, `WISANYME1S21`). `PREMIUM`, `230` and `S` are not codes — matching
on those would be matching on nothing.

### The rule that makes it safe

> **Several candidates are accepted only when every one of them publishes the same
> rated capacity. Fields that disagree are dropped, never chosen.**

This is what unlocks "one outdoor unit, several packages" — and it is also the
guard. If a shared code turned out **not** to determine capacity (a tank code, say),
the candidates carrying it would disagree, and the record is refused rather than
guessed. The protection is the unanimity, not our belief about which code is the ODU.

Hard rejections: a different refrigerant, a different electrical phase, or — where
identity rests on the **name** — different numeric tokens.

### Why fuzzy matching may never accept

`ecoGEO+ B1 230 **1-6** PRO` (a 1–6 kW unit) is **90 % text-similar** to
`ecoGEO+ B1 230 **4-16** PRO` (16 kW). Auto-accepting on similarity would have put
16 kW on a 6 kW product and moved it into the wrong segment. Numbers in a model name
usually *are* the product.

### Source disagreement is not identity disagreement

The registry publishes a measured rated output; EPREL publishes the label's design
load. The same Samsung unit reads **5.5 kW** in one and **6.0 kW** in the other.
That is two quantities, not two products — so the project's existing rule decides:
one performance source per record, registry first. It is never averaged, and the
larger is never taken.

## Results (2026-06 PEL snapshot, 2026-07 registry + EPREL)

| Outcome | Records |
|---|---:|
| Accepted — exact model (Tier A) | 79 |
| Accepted — component identity (Tier A) | 136 |
| Accepted — component identity (Tier B) | 187 |
| Accepted — shared unit code (Tier B) | 176 |
| **Recovered by this pass** | **578** |
| Additionally recovered by the primary matchers, via the 3 new brand aliases | 33 |
| **Total recovered** | **611** |
| Review queue (Tier C — human, never auto-applied) | 502 |
| Rejected — candidates disagree on capacity | 128 |
| No shared product identity | 1,208 |
| Brand absent from every European source | 424 |

Segments, before → after: residential **1,609 → 2,216**, commercial **2,013 → 2,016**,
unclassified **2,745 → 2,134**.

### Manufacturer normalization

Every unclassified record's brand was already in
`scripts/ofgem/manufacturer-short-names-gb.json` — the gap was never PEL-side
normalization. Three PEL brands were aligned to the European registry's short name,
each on **product-code evidence, not name similarity**:

| PEL brand | Registry brand | PEL rows sharing an exact product code |
|---|---|---|
| Guangdong New Energy Technology Co., Ltd. | GD New Energy | 22 of 26 |
| Guangdong Answer Technology Co., Ltd | Answer | 22 of 22 |
| Foshan Goodheat Technology Ltd. | Goodheat | 5 of 5 |

**Left unresolved on purpose:** `Hisense Hitachi` (19 rows) — a joint-venture brand.
Only 4 of its rows share a code with Hisense, and merging distinct brands because
they share a corporate group is exactly what the rules forbid.

## What the match may and may not do

It transfers **technical data only**. It never makes a product PEL-listed, never
implies BUS eligibility, and never names a source country in the UK UI. Provenance
stays internal: the recovered record carries `bafa_reference_*` or `eprel_*` fields
with the method and tier encoded in the match type (`shared_component:B`).

Hardware matched to a registry record is also excluded from the "Not on PEL"
European catalogue, so the same product is never counted twice.
